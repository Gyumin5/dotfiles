# /plan: 설계 우선 워크플로우

코드 작성 전에 설계 문서를 먼저 만들고, 사용자 승인 후 구현한다.

## 워크플로우

### 1. 요구사항 분석

사용자의 요청을 분석하여:

- 핵심 목표 정리
- 영향 범위 파악 (어떤 파일/모듈이 변경되는지)
- 제약 조건 확인 (CLAUDE.md의 Do not 섹션 등)

### 2. 설계 문서 작성

아래 형식으로 설계를 제시:

```markdown
## 설계: [작업 제목]

### 목표

(무엇을 달성하려는가)

### 접근 방식

(어떻게 구현할 것인가 - 2-3줄)

### 변경 계획

1. `path/to/file.ts` - 무엇을 변경
2. `path/to/new.ts` - 새로 생성 (이유)
3. ...

### 고려 사항

- 대안 A vs B → B 선택 (이유)
- 리스크: ...

### 완료 기준

- [ ] 빌드 성공
- [ ] 테스트 통과
- [ ] ...
```

### 3. 사용자 승인

- 설계를 보여주고 승인/수정 요청을 받는다
- **승인 전에 코드를 작성하지 않는다**
- 수정 요청이 있으면 설계를 업데이트

### 4. 구현

승인 후:

- 설계에 따라 순서대로 구현
- 각 단계 완료 시 짧게 보고
- 설계에서 벗어나야 할 경우 사용자에게 알림

## 규칙

- 설계는 간결하게 (50줄 이내)
- 파일 목록은 구체적으로 (경로 포함)
- 사소한 작업(파일 1개 수정 등)에는 과도한 설계 불필요 → 간단히 "X를 Y로 변경합니다" 정도로
- 대규모 작업에서만 전체 형식 사용
