# /plan

코드 작성 전 설계 → 사용자 승인 → 구현.

## 절차

1. 요구사항 분석: 핵심 목표 / 영향 범위 / 제약 조건 (CLAUDE.md Do not 등)
2. 설계 제시 (아래 형식)
3. 사용자 승인 받기. 승인 전에 코드 작성 금지. 수정 요청 있으면 설계 업데이트
4. 승인 후 순서대로 구현. 각 단계 짧게 보고. 설계에서 벗어나야 하면 사용자에게 알림

## 형식
```markdown
## 설계: [제목]
### 목표
(무엇을 달성)
### 접근
(어떻게 - 2~3줄)
### 변경 계획
1. `path/file.ts` - 무엇 변경
2. `path/new.ts` - 새로 생성 (이유)
### 고려 사항
- 대안 A vs B → B 선택 (이유)
- 리스크: ...
### 완료 기준
- [ ] 빌드 성공 / 테스트 통과 / ...
```

## 규칙
- 설계 50줄 이내, 파일 경로 구체적으로
- 사소한 작업(파일 1개 수정)은 한 줄 요약으로 충분 ("X를 Y로 변경")
- 대규모 작업에서만 전체 형식
